# RAG 시스템 허접 버전 만들 때 고려할 사항

## 한눈 체크리스트

아래만 챙겨도 "대충 돌아가는 RAG"에서 "최소한 덜 위험한 RAG"로 갈 수 있다.

- `무슨 질문`에 답할 시스템인지 먼저 정한다.
- `어떤 문서`를 넣을지 정하고, 신뢰 가능한 출처만 쓴다.
- 원문을 그대로 넣지 말고 `텍스트 추출 품질`을 확인한다.
- 청킹은 너무 잘게 자르지 말고 `문맥이 남는 크기`로 자른다.
- 검색은 처음부터 복잡하게 가지 말고 `벡터 검색 + top-k`부터 시작한다.
- 답변할 때는 검색 결과가 없으면 `모른다고 말하게` 만든다.
- 답변에 `출처`를 같이 보여준다.
- "잘 되는 것 같은데?"로 끝내지 말고 `샘플 질문셋`으로 테스트한다.
- `환각`, `오답`, `근거 없음`을 따로 본다.
- 처음 버전은 `작게`, `빨리`, `측정 가능하게` 만든다.

---

## 1. 제일 먼저 정할 것

RAG는 생각보다 "검색"보다 "범위 설정"이 더 중요하다.

### 꼭 정해야 하는 질문

- 이 시스템은 어떤 질문에 답해야 하나?
- 어떤 질문은 답하면 안 되나?
- 문서 범위는 어디까지인가?
- 최신성이 중요한가, 아니면 고정 문서 기반인가?

### 예시

- 좋은 범위: "사고 과실비율 기준 PDF를 바탕으로 과실비율 관련 질문에 답한다"
- 나쁜 범위: "자동차 사고 관련 모든 질문에 다 답한다"

범위가 넓을수록 검색 품질보다 환각 문제가 먼저 터진다.

---

## 2. 문서 수집에서 생기는 문제

RAG 성능은 모델보다 문서 상태에 더 크게 흔들린다.

### 체크포인트

- 원문 파일이 신뢰 가능한가?
- 같은 내용의 중복 문서가 많은가?
- 스캔 PDF인가, 텍스트 PDF인가?
- 표, 각주, 머리말, 페이지 번호가 섞여서 노이즈가 생기지 않는가?
- 문서 버전이 여러 개인가?

### 실무적으로 중요한 점

- PDF에서 텍스트가 이상하게 뽑히면 뒤는 거의 다 무너진다.
- 페이지 번호, 머리글/바닥글, 목차 같은 반복 텍스트는 검색 노이즈가 된다.
- 오래된 문서와 최신 문서가 같이 들어가면 답변 충돌이 난다.

### 최소 대응

- 텍스트 추출 결과를 몇 페이지 직접 본다.
- 불필요한 반복 문자열은 제거한다.
- 문서별로 `source`, `page`, `version` 메타데이터를 남긴다.

---

## 3. 청킹은 왜 중요한가

청킹이 이상하면 검색도 이상하고, 답변 근거도 흐려진다.

### 너무 작은 청크의 문제

- 문맥이 끊긴다.
- 정의와 예외가 분리된다.
- 검색은 됐는데 답변이 애매해진다.

### 너무 큰 청크의 문제

- 관련 없는 내용이 같이 들어간다.
- 임베딩이 뭉개진다.
- 토큰 낭비가 커진다.

### 허접 버전 추천

- 먼저 `문단 단위` 또는 `500~1000자 내외`로 시작
- `100~200자` 정도 overlap 고려
- 표/조항/문단 번호가 중요하면 그 경계를 최대한 보존

핵심은 "사람이 읽었을 때 의미가 끊기지 않는 단위"다.

---

## 4. 임베딩과 벡터 DB

처음 버전은 여기서 멋부릴 필요가 거의 없다.

### 추천 원칙

- 임베딩 모델은 한국어 문서를 어느 정도 처리 가능한 것으로 고른다.
- 저장소는 처음엔 `Chroma` 또는 `FAISS` 정도면 충분하다.
- 나중에 문서가 커지면 Milvus, Elasticsearch, Pinecone 등을 본다.

### 체크포인트

- 문서 언어와 임베딩 모델 언어 특성이 맞는가?
- 같은 모델로 임베딩한 데이터와 검색 쿼리를 일관되게 쓰는가?
- 메타데이터 필터링이 필요한가?

### 주의

- 임베딩 모델 바꾸면 재색인 필요할 가능성이 높다.
- 데이터가 적은데 DB부터 복잡하게 가면 개발 속도만 느려진다.

---

## 5. 검색 전략

처음부터 하이브리드 검색, reranker, query expansion을 다 넣지 않아도 된다.

### MVP 추천 순서

1. 벡터 검색
2. `top-k=3~5` 정도로 시작
3. 결과를 LLM에 넣어 답변
4. 부족하면 그때 BM25나 reranker 추가

### 체크포인트

- 검색된 문서가 진짜 질문과 관련 있는가?
- top-k가 너무 작아서 놓치지 않는가?
- top-k가 너무 커서 잡음이 늘어나지 않는가?

### 나중에 추가해볼 것

- `BM25 + 벡터 검색` 하이브리드
- reranker
- query rewrite
- metadata filter

---

## 6. 프롬프트에서 꼭 막아야 할 것

RAG는 검색했다고 끝이 아니라, 답변 규칙을 잘 박아야 한다.

### 최소 프롬프트 규칙

- 주어진 문맥 안에서만 답변한다.
- 근거가 부족하면 모른다고 말한다.
- 추측하지 않는다.
- 가능하면 출처를 같이 적는다.

### 예시 규칙

> 제공된 문맥에서 확인되는 내용만 답변하라.  
> 문맥에 없으면 모른다고 답하라.  
> 답변 마지막에 참고한 페이지 또는 출처를 표시하라.

### 흔한 실패

- 검색 결과가 부정확한데 모델이 그럴듯하게 메운다.
- 여러 청크가 충돌하는데 하나로 섞어서 확신 있게 말한다.
- 질문이 문서 범위 밖인데도 일반 상식으로 답한다.

---

## 7. 출처 표시는 거의 필수

허접 버전일수록 출처를 보여줘야 사용자가 시스템을 덜 오해한다.

### 왜 필요한가

- 답변 근거를 검증할 수 있다.
- 잘못된 검색 결과를 빨리 찾을 수 있다.
- 사용자가 "이 답이 문서 기반인지" 알 수 있다.

### 최소 구성

- 문서명
- 페이지 번호
- 가능하면 관련 원문 일부

예:

```text
답변: ...
출처: 230630_자동차사고 과실비율 인정기준_최종.pdf, p.14
```

---

## 8. 평가를 꼭 따로 봐야 한다

RAG는 데모에서는 잘 되는 것 같아도 실제 질문에서 갑자기 망가진다.

### 최소 평가셋 만들기

- 자주 나올 질문
- 애매한 질문
- 문서에 없는 질문
- 비슷한 표현의 질문
- 숫자/조건/예외가 섞인 질문

### 평가할 항목

- 검색 적중 여부
- 답변 정확도
- 출처 정확도
- 문서 밖 질문에 대해 잘 거절하는지

### 아주 단순한 기준 예시

- `정답`: 문서 근거와 일치
- `부분 정답`: 일부만 맞음
- `오답`: 근거와 다름
- `환각`: 문서에 없는 내용 생성
- `검색 실패`: 관련 청크를 못 찾음

---

## 9. 운영 관점에서 미리 볼 리스크

작은 프로젝트여도 아래는 빨리 문제 된다.

### 데이터 리스크

- 문서 업데이트 후 재색인을 안 함
- 구버전/신버전 혼재
- OCR 오류 방치

### 서비스 리스크

- 응답 속도 느림
- 토큰 비용 증가
- 긴 문맥 때문에 모델 품질 저하

### 제품 리스크

- 사용자가 "이건 무조건 맞는 시스템"이라고 믿음
- 문서 기반 답변과 일반 추론 답변의 경계가 불명확함

### 최소 대응

- 문서 버전 관리
- 색인 시각 기록
- 답변 화면에 "문서 기반 참고용" 성격 표시

---

## 10. 처음 버전에서 굳이 안 해도 되는 것

아래는 나중으로 미뤄도 된다.

- 멀티 에이전트
- 복잡한 라우팅
- 여러 벡터 DB 동시 운영
- 고급 평가 자동화
- 지나치게 세밀한 chunk 전략
- 과한 프롬프트 튜닝

처음엔 "작동", "검증", "개선 포인트 확인"이 더 중요하다.

## 11. 이 프로젝트에서 제일 단순하게 시작하는 방법

현재 repo 기준으로는 PDF 문서 기반 QA가 가장 자연스럽다.

### 추천 구성

1. PDF 텍스트 추출
2. 페이지별 또는 문단별 정리
3. 청킹
4. 임베딩 생성
5. `Chroma` 또는 `FAISS`에 저장
6. 질문 시 top-k 검색
7. 검색 결과 + 출처를 붙여 LLM에 전달
8. 답변과 출처를 함께 반환

### 아주 단순한 시스템 그림

```text
PDF -> 텍스트 추출 -> 청킹 -> 임베딩 -> 벡터 저장소
사용자 질문 -> 임베딩 -> 유사도 검색 -> 관련 청크 -> LLM 답변
```

### 이 단계에서 꼭 넣을 것

- 청크마다 `source`, `page` 저장
- 검색 결과 원문 일부 로그 확인
- 문맥 없으면 "모르겠습니다" 처리
- 샘플 질문 10개 정도로 수동 테스트

---

## 12. 추천 개발 순서

1. 문서 1개로 시작
2. 텍스트 추출 품질 확인
3. 청킹 결과 직접 보기
4. 검색 결과만 먼저 확인
5. 그다음 LLM 답변 연결
6. 마지막에 UI 붙이기

UI보다 검색 품질 확인이 먼저다.

---

## 13. 한 줄 요약

허접 버전 RAG의 핵심은 "좋은 모델"이 아니라 아래 4개다.

- 문서가 제대로 뽑히는가
- 청크가 의미 있게 잘리는가
- 검색이 맞는 문맥을 찾는가
- 문맥이 없을 때 모른다고 말하는가

이 4개만 잡아도 첫 버전 품질이 꽤 달라진다.
